PROBLEM

Current Architecture & Its Problem
Right now your RAG chatbot works like this:

This works, but has serious production concerns:

API key exposed — VITE_GOOGLE_AI_API_KEY is baked into the client bundle. Anyone can extract it from browser DevTools and use your Gemini quota.
Rate limiting is fake — requestTimestamps in ragService.js and inventoryRagService.js live in browser memory. Refreshing the page resets them. Opening a second tab bypasses them entirely.
No cost control — A malicious user could call Gemini thousands of times on your dime.

----

The Fix: Move RAG Logic Server-Side
You need a server-side layer between the browser and Gemini. The question is: Cloud Functions or Cloud Run?

Decision Framework
Factor	Cloud Functions (2nd gen)	Cloud Run
Max request timeout	60 min (2nd gen)	60 min
HTTP streaming (SSE)	✅ Supported (2nd gen)	✅ Supported
Cold starts	~1-3s	~1-3s (similar, both use Cloud Run under the hood)
Firestore integration	Native (firebase-admin)	Native (firebase-admin)
Scaling	Auto (0 to N)	Auto (0 to N)
Complexity	Low (just write functions)	Medium (need a small Express/Hono server)
Firebase Auth verification	Built-in with onCall / easy with middleware	Manual (verify ID token yourself)
Cost at low traffic	Scale to zero ✅	Scale to zero ✅ (min instances = 0)
Your use case duration	~5-30s (Firestore read + Gemini stream)	Same
Already in your stack	✅ Firebase project exists	Would need setup

My Recommendation: Cloud Functions (2nd gen) only
Your chatbot request lifecycle is:

Firestore read (~200ms) → Build context (~5ms) → Gemini stream (~5-25s)

This is a single short-lived request-response with streaming. You don't need a persistent server, custom routing, or long-running containers. Cloud Functions 2nd gen (which runs on Cloud Run under the hood anyway) gives you everything you need with less setup.

You'd only need standalone Cloud Run if you had:

Complex multi-route API (you don't — you have 3-4 endpoints max)
Long-running background jobs >60 min (you don't)
Custom binary dependencies or GPU needs (you don't)
Need for WebSocket (you don't — HTTP streaming is sufficient as we established)

V.V.I: 
Summary
Question	                    Answer
HTTP streaming or WebSocket?	HTTP streaming (SSE) — your request is always request→stream-response, never server-initiated
Cloud Functions or Cloud Run?	Cloud Functions (2nd gen) — simpler, fits your 5-30s request lifecycle, scales to zero, and is already Cloud Run under the hood
What moves server-side?	        API key, rate limiting, Firestore reads for RAG context, Gemini calls
What stays client-side?	        Auth token acquisition, SSE reader, UI rendering
