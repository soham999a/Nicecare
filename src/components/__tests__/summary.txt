The changes I made provide several important benefits:

1. Accessibility (a11y) Compliance
Adding htmlFor/id attributes to labels and inputs allows:

Screen readers to properly announce form fields to visually impaired users
Clicking on labels to focus the associated input (better UX)
Compliance with WCAG guidelines (Web Content Accessibility Guidelines)

```
// Before: Label and input not connected
<label className="label">Name *</label>
<input name="name" ... />

// After: Properly associated
<label className="label" htmlFor="name">Name *</label>
<input id="name" name="name" ... />
```

2. Testability
The getByLabelText() query from Testing Library only works when labels are properly associated:

```
// This test now works:
const nameInput = screen.getByLabelText(/name/i);
```

This is the recommended way to query form elements because it tests what users actually interact with.

3. Better User Experience
aria-label on the theme toggle button tells screen readers what the button does instead of just reading "moon emoji"
aria-label="Close" on the modal close button provides context instead of just "×"
4. Robust Tests
The test fixes ensure:

Tests don't fail due to duplicate element names (scoped queries)
Tests match actual component behavior (number inputs return numbers)
Tests work with browser validation (using valid email formats)

These are best practices that make your app:

✅ Accessible to all users
